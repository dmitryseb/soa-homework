# Marketplace Architecture Homework

## 1. Введение
Проектируетcя базовая микросервисная архитектура маркетплейса без бизнес-логики.
Реализован и засунут в контейнер 1 сервис: `User Service` с health-check endpoint.

## 2. C4 Container Diagram
Диаграмма: `homework-1/c4-container.mmd`

Контейнеры в системе:
- `API Gateway (клиентский backend)` - единая точка входа для Покупателя/Продавца.
- `User Service` - учетные записи и роли (покупатель/продавец).
- `Catalog Service` - товары и управление каталогом.
- `Feed Service` - выдача персонализированной ленты.
- `Order Service` - создание и жизненный цикл заказа.
- `Payment Service` - расчеты и интеграция с платежным провайдером.
- `Notification Service` - уведомления о статусах заказа.
- `Event Broker` - асинхронный обмен событиями.

## 3. Домены и ответственность
- `User domain`: регистрация, профили, роли, базовый статус пользователя.
- `Catalog domain`: карточки товаров, остатки, цены, видимость товара.
- `Feed domain`: формирование ленты, простая персонализация (по категории/истории просмотров).
- `Order domain`: корзина/чекаут, статус заказа, связь заказа с пользователем и товарами.
- `Payment domain`: платежные транзакции, статусы оплаты, учет попыток/ошибок.
- `Notification domain`: отправка email/sms/push о статусах заказа и оплаты.

## 4. Распределение доменов по сервисам
- `User domain` -> `User Service`.
- `Catalog domain` -> `Catalog Service`.
- `Feed domain` -> `Feed Service`.
- `Order domain` -> `Order Service`.
- `Payment domain` -> `Payment Service`.
- `Notification domain` -> `Notification Service`.

Логика разбиения:
- высокая cohesion внутри сервиса (одна бизнес-зона ответственности);
- низкий coupling между сервисами (контракты API + события);
- независимое развертывание и масштабирование (например, `Feed Service` масштабируется отдельно от `Payment Service`).

## 5. Границы владения данными
Каждый сервис владеет своей БД, общих БД нет:
- `User Service` -> `User DB`.
- `Catalog Service` -> `Catalog DB`.
- `Feed Service` -> `Feed DB` (read-model для быстрой выдачи).
- `Order Service` -> `Order DB`.
- `Payment Service` -> `Payment DB`.
- `Notification Service` -> `Notification DB`.

## 6. Взаимодействия сервисов
Синхронные (HTTP):
- `Gateway -> User/Catalog/Feed/Order` (основные API-вызовы собраны в одну группу).
- `Order -> Payment`.
- `Feed -> Catalog` (получение данных каталога для выдачи).

Асинхронные (через broker):
- `OrderCreated` публикует `Order Service`.
- `Payment` публикует `Payment Service`.
- `Notification Service` и `Feed Service` подписываются на события.

## 7. Варианты декомпозиции
### Вариант A (финальный): 6 доменных сервисов + broker
Плюсы:
- четкие границы доменов и владения данных
- независимый релиз и масштабирование сервисов
- проще эволюционировать персонализацию ленты

Минусы:
- выше операционная сложность
- больше сетевых взаимодействий
- нужна дисциплина контрактов и наблюдаемость

### Вариант B: модульный монолит + выделенные Payment и Notification
Плюсы:
- проще старт разработки и локальная отладка
- ниже инфраструктурные затраты на старте
- меньше проблем с распределенными транзакциями

Минусы:
- хуже независимое масштабирование
- риск роста в распределенный монолит при дальнейшем выносе
- медленнее time-to-market для независимых команд

## 8. Почему выбран вариант A
Выбран `Вариант A`, потому что:
- требования прямо покрывают несколько независимых доменов (catalog/order/payment/feed/notification)
- есть естественные границы данных (особенно `Payment`)
- нужен потенциал масштабирования без избыточной сложности (взята минимальная версия сервисов и один broker)

При этом реализация остается базовой: без бизнес-логики, только архитектурный каркас.

## 9. Запуск User Service в Docker

Для запуска ввести команды:
```bash
cd homework-1
docker compose up --build -d
curl -i http://localhost:8080/health
```

Ожидаемый результат:
- HTTP статус `200 OK`
- JSON с полем `status: "ok"`
