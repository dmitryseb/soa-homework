# Architecture Decisions

## ADR-001: Микросервисная декомпозиция по доменам

Решение:
- разделили систему на `User Service`, `Catalog Service`, `Feed Service`, `Order Service`, `Payment Service`, `Notification Service`

Почему так:
- у каждого сервиса своя понятная зона ответственности
- сервисы меньше зависят друг от друга, потому что общаются через API и события
- каждый сервис можно выпускать и масштабировать отдельно

## ADR-002: Отдельная БД на сервис

Решение:
- у каждого сервиса своя БД, общей базы нет

Почему так:
- сразу понятно, кто владеет какими данными
- меньше риск, что изменение в одном сервисе сломает другой
- проще разворачивать и обновлять сервисы независимо

## ADR-003: Гибрид взаимодействий (sync + async)

Решение:
- для обычных запросов используем HTTP
- для статусов заказа/оплаты и уведомлений используем события через broker

Почему так:
- API-запросы остаются простыми и понятными
- сервисы меньше завязаны друг на друга во времени
- можно добавлять новых подписчиков на события без переделки текущих сервисов

## ADR-004: Базовый сервис для демонстрации инфраструктуры

Решение:
- для ДЗ реализован только `User Service` с endpoint `GET /health`.

Почему так:
- по заданию бизнес-логика пока не нужна
- этого достаточно, чтобы показать рабочую контейнеризацию и запуск
- можно быстро проверить, что все поднимается и отвечает `200 OK`
